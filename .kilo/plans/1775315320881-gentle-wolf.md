# Plan for Phase 3 - Coin Intelligence Layer

## Overview
Phase 3 focuses on creating the Coin Intelligence Service to provide context about coins before analysis. This involves building a new service file `backend/src/services/coinIntelligence.service.ts` that aggregates data from various sources with caching.

## Dependencies
- **Phase 1**: Database tables must exist (including `coinIntelligenceCache`).
- **Phase 2**: Data services must be built (priceService, binanceHistory.service, wikipedia.service).
- **Verification**: Confirm that the required models, services, and database schema are in place before starting implementation.

## Implementation Plan

### Step 1: Review Dependencies
- Use file search tools to verify existence of:
  - `backend/src/models/market.model.ts` (contains `coinIntelligenceCache` table)
  - `backend/src/services/priceService.ts` (with `getPriceWithFallback` and `PriceResult`)
  - `backend/src/services/binanceHistory.service.ts` (with `getBinanceHistory` and `BinanceHistoryResult`)
  - `backend/src/services/wikipedia.service.ts` (with `getWikipediaBackground`)

### Step 2: Create the Service File
- Create new file: `backend/src/services/coinIntelligence.service.ts`
- Implement the `CoinIntelligence` interface exactly as specified
- Implement `getCoinIntelligence` function with:
  - Cache check using Drizzle ORM (4-hour TTL)
  - Parallel data fetching with `Promise.allSettled`
  - Intelligence object construction
  - DB upsert with conflict handling

### Step 3: Ensure Type Safety
- Use only specified types, no `any`
- Properly cast types where needed (e.g., `trend8w`, `dataSource`)
- Handle `priceChange30d` parsing for DB storage

### Step 4: Testing and Verification
- Manual test: Call `getCoinIntelligence('SOL')` twice to verify caching
- Check DB for proper upsert and timestamp updates
- Ensure no crashes on individual service failures
